# v0.2.0 macOS release preparation

This document is a release-candidate checklist, not a release announcement or evidence that any
artifact has been published.

## Current status

As of 2026-07-18:

- source and bundle metadata use version `0.2.0`;
- the native macOS app has a host-architecture local acceptance path;
- that local path is sandboxed and ad-hoc signed only;
- no v0.2.0 release tag is represented as created;
- no v0.2.0 DMG, release manifest, or checksum file is represented as published; and
- no Developer ID signing or Apple notarization result is claimed.

The README and platform guides must keep this state visible until every gate below is completed and
the distribution state is updated deliberately.

## Intended user artifacts

A complete macOS release set will contain separate native packages rather than one combined helper:

```text
TV-Time-Backup-Extractor-0.2.0-macOS-Apple-Silicon-arm64.dmg
TV-Time-Backup-Extractor-0.2.0-macOS-Intel-x86_64.dmg
release-manifest-arm64.json
release-manifest-x86_64.json
SHA256SUMS
```

Apple silicon users select the arm64 DMG; Intel users select the x86_64 DMG. PyInstaller helper
executables must never be merged with `lipo`. The release pipeline freezes and validates each helper
and Swift executable for one exact architecture.

The source archive automatically generated by a hosting platform is not a substitute for either
signed application package. A Python wheel or source distribution may be published separately only
after its own clean-environment build, install, dependency, privacy, and checksum gates pass.

## Prerequisites

Release packaging requires all of the following on a controlled Apple silicon build Mac:

- a full Xcode installation selected with `xcode-select` and a Swift 6.2-capable toolchain;
- Rosetta 2 so the x86_64 build Python can execute on Apple silicon;
- the official python.org CPython 3.13.12 universal2 executable, which must run natively as both
  `arm64` and `x86_64` and match the repository's reviewed release-native profile exactly;
- the exact pinned runtime and macOS-build dependencies;
- an available **Developer ID Application** identity;
- a working `notarytool` Keychain profile containing the Apple notarization credentials;
- access to Apple's notarization service; and
- enough private local space for two clean architecture build trees and their DMGs.

Do not put signing keys, Apple credentials, backup passwords, or notary secrets in the repository,
workflow logs, command history, environment files, or release notes. `TVTIME_NOTARY_PROFILE` names a
preconfigured Keychain profile; it is not the credential itself.

## Pre-release source gates

Freeze the intended source commit and confirm a clean review scope. Before running the distribution
script, complete:

```text
python -m ruff check .
python -m ruff format --check .
PYTHONWARNINGS=error::ResourceWarning PYTHONDONTWRITEBYTECODE=1 \
  python -m unittest discover -s tests -v
python -I script/build_python_distributions.py \
  --source-commit "$(git rev-parse HEAD)" --outdir dist-python-release
python -I script/verify_python_release.py \
  --root dist-python-release \
  --source-commit "$(git rev-parse HEAD)" \
  --source-tree "$(git rev-parse 'HEAD^{tree}')"
python -I script/sdist_metadata.py --verify dist-python-release/*.tar.gz
xcrun swift-format lint --strict --recursive macos/Package.swift macos/Sources macos/Tests
swift test --package-path macos --disable-automatic-resolution
swift test --package-path macos --disable-automatic-resolution --configuration release
swift build --package-path macos --disable-automatic-resolution --configuration release
git diff --check
```

Also require:

- a fresh temporary environment with `requirements.lock` installed using `--require-hashes` and
  `--only-binary=:all:`, `requirements-source-build.lock` installed the same way, the project
  installed with `--no-index --no-build-isolation --no-deps`, a passing `pip check`, and all CLI
  help screens exercised;
- full-recovery Python CI on macOS/Linux plus Windows existing-extraction and fail-closed contract
  CI at the supported boundary versions;
- native macOS format, debug/release test, and optimized-build CI;
- shell syntax and Python packaging-helper compilation checks;
- dependency/license review for all requirement inputs and their generated hash locks; and
- a repository-wide privacy scan for recovery artifacts, credentials, private paths, and generated
  output.

No real backup or recovery output may enter tests, CI, build staging, notarization submissions,
release manifests, checksums, screenshots, documentation, or downloadable assets.

An authorized local acceptance operator may run `script/validate_recovery_output.py` with the
private completed-output path supplied only over standard input. The fixed gate/result lines and
aggregate counts are local review aids, not CI or release inputs; do not retain or publish them.

`script/build_release_app.sh` verifies the clean reviewed commit, prepares a safe read-only
`git archive` beneath the ignored distribution staging root, and re-executes the committed release
builder from that immutable tree. It verifies exact staged membership and content again before
publishing local artifacts. Checkout edits—including a change-then-restore race—are never compiled
or copied into the signed app.

## Build command

After the prerequisites and source gates pass, run the fail-closed release script with real local
configuration values:

```text
TVTIME_SIGNING_IDENTITY="Developer ID Application: ORGANIZATION (TEAMID)" \
TVTIME_NOTARY_PROFILE="KEYCHAIN_PROFILE_NAME" \
TVTIME_BUILD_PYTHON="/path/to/universal2/python3" \
TVTIME_RELEASE_COMMIT="FULL_REVIEWED_40_CHARACTER_GIT_COMMIT" \
TVTIME_RELEASE_VERSION="0.2.0" \
  ./script/build_release_app.sh
```

`TVTIME_RELEASE_COMMIT` must be the exact full commit that passed review and CI. The script requires
that commit to be `HEAD`, requires a completely clean tracked and untracked worktree, requires the
committed `macos/Package.resolved`, and records both the commit and its Git tree in each manifest.

The script also refuses a missing or non-Developer-ID identity, unavailable identity, missing
notary profile name, non-universal build Python, any build Python other than exact CPython 3.13.12,
a bundled OpenSSL/mpdecimal/SQLite tuple that does not match the reviewed official release profile,
absent full Xcode, missing Rosetta execution, version mismatch, running recovery process, unsafe
generated path, held build lock, or existing final release directory.

It builds into a confined temporary stage beneath the generated distribution directory and promotes
only the completed output set. A failed stage is preserved for local diagnosis and must not be
published or allowed to contain private recovery data.

## Automated artifact gates

For each architecture, the script must complete all of these without bypasses:

1. Build the Swift executable for the exact target triple with automatic dependency resolution
   disabled, and prove `Package.resolved` did not change.
2. Rebuild the Python helper and its environment from a fresh stage under that architecture, using
   the exact transitive hash lock with `--require-hashes`; release builds never accept a cached
   helper or cached build environment.
3. Verify every Mach-O in the app/helper tree has only the intended architecture.
4. Select the one exact source-controlled native-license profile matching the executing CPython and
   the bundled OpenSSL, mpdecimal, and SQLite runtime versions. A release must select
   `official-cpython-3.13.12-universal2`; the local Homebrew profile is deliberately not
   release-eligible.
5. Include project, CPython, dependency, and third-party notices. Inventory every non-system Mach-O
   path, architecture, primary component, any statically embedded components, exact component
   versions, and exact shipped license texts; reject every unmapped, unacknowledged composite,
   missing, extra, cross-mapped, or hash-mismatched item.
   Record the SHA-256 of a private copy canonicalized with
   `/usr/bin/codesign --remove-signature`, require stable equality before and after signing, and
   never modify the source binary.
6. Remove extended attributes and lint both bundle plists and entitlement files.
7. Scan the complete bundle for forbidden recovery filenames, private build paths, broken links, and
   links that escape the bundle.
8. Sign nested code inside-out with the hardened runtime and least-privilege sandbox entitlements.
9. Re-inventory the signed app and require the complete native path/architecture/component/version/
   license/canonical-hash mapping to match the pre-sign manifest exactly. Then verify deep/strict
   signatures, assert the hardened-runtime flag on every Mach-O and nested code
   container, and verify the exact production entitlements. The production helper profile must not
   contain the local ad-hoc build's `com.apple.security.cs.disable-library-validation` exception.
10. Before notarization, copy the already Developer ID-signed helper bundle to a private temporary
   directory, re-sign only that copy's outer helper bundle for direct shell launch with the same
   Developer ID identity, Hardened Runtime, and an exact empty entitlement dictionary, then run the
   synthetic protocol preflight under the package's exact architecture. The gate rejects ad-hoc or
   mismatched identities, Team IDs, architectures, entitlements, signatures, and runtime flags. It
   never adds the local library-validation exception and never modifies the app being notarized.
11. Submit the app to Apple, wait for successful notarization, staple the ticket, validate the staple,
   and pass Gatekeeper execution assessment.
12. Create, sign, notarize, staple, and Gatekeeper-assess the architecture-specific DMG.
13. Generate a per-architecture provenance manifest containing the reviewed source commit, source
    tree, Swift lock digest, Python dependency-lock digest, and installed dependency inventory.
14. Verify the mounted final DMG repeats the exact native-license gate, scan the final output tree,
    and generate `SHA256SUMS` covering both DMGs and both manifests.

The script performs no network upload to GitHub or another distribution service. Successful local
packaging is necessary but not sufficient for publication.

The human-readable `requirements*.txt` files are review inputs. Install and package only from the
corresponding `requirements*.lock` files. Every lock contains exact transitive pins and allowed wheel
hashes; CLI CI installs them with `--require-hashes` on macOS, Windows, and Linux at every supported
Python version from 3.10 through 3.13. Native-helper CI and packaging use exact CPython 3.13.12.
The source-build backend has its own minimal lock, and CI disables both the index and build isolation
while installing the local project so pip cannot create a network-resolved build environment.
The source-distribution builder rewrites tar ownership to numeric zero with blank owner/group names,
then the separate metadata verifier rejects any local UID, GID, user name, or group name.

`macos/Bundle/NativeLicenses/PROVENANCE.json` is the controlled allowlist for native-runtime license
sources. Each profile binds the exact CPython license and native-runtime versions to an upstream
release URL, archive SHA-256, license path, controlled extraction method, and exact checked-in
license-text SHA-256. CPython, OpenSSL, and mpdecimal are byte-exact whole-file copies. SQLite uses
lines 4–9 of the pinned `sqlite3.h`, removes the leading `**` and at most one following space from
each line, joins with LF, and ends with exactly one LF. Builds do not fetch license material or
accept a host-provided CPython-license override. Those archive hashes establish the provenance of
the checked-in **license inputs**; they do not prove the origin of the bundled executable bytes. The
in-bundle native manifest and per-architecture release manifest record the actual packaged binaries,
while the signed DMG and its published checksum bind the downloadable artifact bytes.

## Manual acceptance gates

Before creating a public release, test copies of the final DMGs—not an earlier app bundle:

- verify each checksum from a separate directory;
- test the arm64 DMG on a clean Apple silicon macOS 14-or-later account;
- test the x86_64 DMG on a clean Intel macOS 14-or-later account;
- confirm normal drag-to-Applications installation and launch without bypassing Gatekeeper;
- confirm the app and helper report the intended version and exact architecture;
- confirm sandbox folder pickers can select an individual completed backup and encrypted
  destination;
- run synthetic preflight, cancellation, retry, completion, report-open, and clean-quit cases;
- parse generated synthetic PDFs and prove every expected section token and occurrence count after
  removing only layout-inserted whitespace, while separately rendering pages for clipping/overlap
  review; and
- run one authorized real encrypted-backup acceptance case entirely outside the repository, with the
  phone disconnected, source read-only state independently checked, both completion markers
  validated, copied-file totals reconciled, and Markdown/HTML/PDF fidelity reviewed.

The authorized real-data case must report only pass/fail gates and generic aggregate behavior to the
release decision. Do not preserve its titles, counts, paths, hashes, screenshots, reports, or backup
in the repository or release record.

If PDF is omitted because text fidelity cannot be guaranteed, that is an acceptable successful case
when the app explains the omission, Markdown and HTML are complete, and the report marker records the
decision. Missing or altered text in any generated report is a release blocker.

## Publication sequence

Only after every source, artifact, and manual acceptance gate passes:

1. Reconfirm the exact reviewed commit and version consistency.
2. Re-run the final repository and artifact privacy scans.
3. Create an annotated version tag locally from the exact reviewed commit. Verify that
   `v0.2.0^{commit}` resolves to that commit before any remote mutation.
4. Push only that tag, fetch it back, and verify that the remote peeled tag still resolves to the
   exact reviewed commit.
5. Create the unpublished draft for that existing tag with `gh release create v0.2.0 --draft
   --verify-tag ...`. Never allow release tooling to auto-create or retarget the version tag.
6. Upload only the two DMGs, two manifests, and `SHA256SUMS` produced by the successful final build,
   plus separately verified CLI packages if intentionally included.
7. Download every draft asset into a fresh directory, verify checksums, signatures, notarization,
   stapling, Gatekeeper assessment, architecture, names, and file sizes again.
8. Check that release notes state macOS 14+, the architecture choice, privacy requirements, optional
   PDF behavior, CLI fallback, and known cache-completeness limitations.
9. Immediately before publishing, fetch and re-verify the remote peeled tag and confirm that the
   draft still targets that exact tag and reviewed commit. Publish only when those identities and
   every draft asset exactly match the validated local set.
10. Update README, support status, and changelog from “unreleased” only after publication is
    confirmed.

Do not publish a partial architecture set, replace an asset in place, reuse a failed build stage, or
retroactively change a checksum. If any post-upload verification fails, keep the release unpublished,
remove the invalid draft asset, fix the cause, and rebuild the entire fresh artifact set.
